Seamless Wallet API (4.0.0)

Download OpenAPI specification:

Seamless Wallet integration protocol for game providers and operators.

In the Seamless Wallet implementation, there is no wallet on the Provider's side. The wallet exists only on the Operator's side. The Operator is responsible for storing each user's balance and all transaction history.

Roles:

  • Provider — Game provider
  • Operator — Game operation handler

Protocol: JSON over HTTPS POST

Signature

All requests and responses are signed via the X-Signature HTTP header — an HMAC-SHA256 hash of the raw request/response body ensuring packet integrity and authenticity.

The signature is computed over the raw body bytes as sent over the wire — no JSON parsing, sorting, or field manipulation is needed.

Algorithm:

  1. Read the raw HTTP body as a byte string
  2. Compute HMAC-SHA256 of that byte string using the shared secret key
  3. Send/verify the result as a 64-character lowercase hex string in the X-Signature header

PHP example:

// --- Sending ---
$body = json_encode($payload, JSON_UNESCAPED_SLASHES | JSON_UNESCAPED_UNICODE);
$signature = hash_hmac('sha256', $body, $secretKey);
// Set header: X-Signature: $signature
// Send body: $body

// --- Receiving ---
$rawBody = file_get_contents('php://input');
$expected = hash_hmac('sha256', $rawBody, $secretKey);
$received = $_SERVER['HTTP_X_SIGNATURE'] ?? '';

if (!hash_equals($expected, $received)) {
    // Respond with error code 5 — Invalid signature
}

$payload = json_decode($rawBody, true);

Pseudocode (language-agnostic):

signature = hmac_sha256(raw_body_bytes, SECRET_KEY)
set_header("X-Signature", signature)

Note: The secret key is provided during integration onboarding. Both sides (Provider and Operator) must use the same key. Always use timing-safe comparison (e.g. hash_equals in PHP) when verifying signatures. The raw body must not be modified by middleware (no re-encoding or pretty-printing) between signing and verification.

Idempotency: All wallet transaction endpoints (/bet, /win, /free-spin-bet, /free-spin-win, /tournament-win, /jackpot-win, /refund) are idempotent on transaction_id. If the Operator receives a request with a transaction_id that has already been successfully processed, it must return the original TransactionResponse (with status: true, the original operator-side transaction_id, and the current balance) — not an error. This ensures safe retries over unreliable networks.

Response header: All responses contain an X-Request-Id header — a unique request identifier for quick transaction lookup on the operator's side.

Provider API

Operator → Provider. Game launch and content delivery endpoints hosted by the Game Provider.

During integration onboarding the Provider must supply a base URL (e.g. https://games.provider.com or https://games.provider.com/api/v1). All Provider API paths in this spec are appended to that base URL. The Operator stores this value per provider and uses it for every outgoing request.

Launch a game session

Opens a game in the provider's iframe/page. Called by the operator's frontend to start a real or demo game session.

The game provider must support auto-resize of game content and mobile games.

Example request URL:

https://games.provider.com/launch-game?action=real&game_id=1056&token=704be4er164ef34cc56ccbb64a960868&lobby_url=https%3A%2F%2Fcasino.operator.com%2Flobby&language=uk
query Parameters
action
required
string
Enum: "real" "demo"

Game mode:

  • real — real-money game
  • demo — demo game
game_id
required
string
Examples:
  • game_id=623 - Numeric game id
  • game_id=Valhalla - Game name
  • game_id=21312:slots - Composite id with category

Identifier of the game

token
required
string <= 255 characters

Gaming session token. Required for real game mode only.

lobby_url
required
string <uri>

URL for redirection to the casino lobby when the player exits the game

client_id
string

Unique Operator identifier assigned by the Provider

language
string <= 2 characters

User's language code (ISO 639-1)

Place a win

Credits the win amount to the player's balance.

Important rules:

  • Win with amount = 0 is required for lost bets.
  • Wins without corresponding bets will not be processed.
  • Wins must be repeated until a response with success is received, except when the Operator responds with error code 6 (original transaction not found) — in that case no corresponding bet exists and retrying is pointless.

Refund a transaction

Refunds a previously placed bet or cancels a win, adjusting the player's balance accordingly.

The type field determines what is being refunded:

  • bet — cancels a bet, credits the amount back to the player
  • win — cancels a win, debits the amount from the player
header Parameters
X-Signature
required
string (Signature) = 64 characters ^[0-9a-f]{64}$

HMAC-SHA256 hex digest of the raw HTTP request body using the shared secret key.

Request Body schema: application/json
required
token
required
string <= 255 characters

Gaming session token (real game only)

user_id
required
integer <int64>

Unique player identifier on Operator's side

type
required
string <= 50 characters
Enum: "bet" "win"

Type of the original transaction being refunded: bet — cancels a bet (credits the player), win — cancels a win (debits the player).

game_id
required
string <= 255 characters

Identifier of the game

round_id
required
string <= 255 characters

Round identifier linking bets to wins. Must be unique between rounds.

transaction_id
required
string <= 255 characters

Unique identifier of the transaction

original_transaction_id
required
string <= 255 characters

Transaction ID of the original bet or win being refunded

amount
required
string <decimal> ^\d+\.\d{1,8}$

Refund amount (must be > 0)

currency
required
string (Currency) <= 10 characters

Currency code. ISO 4217 for fiat (3 chars), or a common ticker for cryptocurrencies (e.g. USDT, BTC).

round_finished
boolean
Default: false

Indicates that this is the last transaction in the round. When true, the Operator may finalize the round for reporting and reconciliation.

object (Extensions)

Optional domain-specific custom fields. Providers may include additional metadata relevant to their game vertical (e.g. lottery, scratch cards). The Operator MUST store these fields alongside the transaction and MAY use them for reporting, reconciliation, or regulatory compliance. Unknown fields MUST be preserved as-is.

Known fields (lottery / instant games): | Field | Type | Description | |----------------|---------|--------------------------------------------| | series_id | integer | Series identifier (instant / scratch) | | draw_id | integer | Draw identifier (draw games) | | barcode | string | Human-readable ticket barcode |

Responses

Operator — implements Seamless Wallet API called by the Provider

https://{operatorHost}/refund

Provider — game launch and content delivery

https://{providerHost}/refund
